# my_bilibili 设计文档

## 1. 目标

`my_bilibili` 是一个完全本地运行的个人 B 站订阅管理工具，用来弥补 B 站官方产品缺少“按个人规则给 UP 主分类并查看不同类型未看视频”的问题。

系统目标：
- 手动维护需要关注的 UP 主列表
- 用标签体系对 UP 主做个性化分类
- 抓取这些 UP 主公开可访问的视频信息
- 在本地网页中按标签查看对应的未看视频
- 通过手动标记维护“已看 / 未看”状态
- 所有个人相关数据均保存在本地，避免依赖常用账号

## 2. 设计原则

- 完全本地运行，不依赖云端服务
- 不接入用户常用 B 站账号
- 只抓取手工录入目标的公开页面内容
- 公开视频数据与个人管理数据严格分离
- 第一版聚焦核心闭环，不扩展笔记、推荐、首页聚合流等能力

## 3. 产品形态

系统由三部分组成：

1. 抓取层：根据手工录入的 UP 主主页链接抓取公开视频列表和基础元信息。
2. 本地数据层：保存 UP 标签、视频基础信息、视频已看状态、同步日志等本地数据。
3. 本地网页层：提供标签视图、UP 主管理页、同步状态页，供用户完成浏览、管理和同步操作。

其中，抓取到的公开视频信息属于外部公开数据；标签、已看状态、同步偏好等属于用户个人本地数据，两者需要在模型和实现上分层。

## 4. 核心数据模型

### 4.1 UP 主

用于保存手工维护的抓取目标。

建议字段：
- `id`
- `uid` 或其他主页唯一标识
- `name`
- `profile_url`
- `enabled`
- `last_sync_at`
- `created_at`
- `updated_at`

### 4.2 标签

标签是用户的本地分类体系，不依赖 B 站。

建议字段：
- `id`
- `name`
- `color`（可选）
- `created_at`
- `updated_at`

### 4.3 UP-标签关联

用于表达一个 UP 主可属于多个标签。

建议字段：
- `up_id`
- `tag_id`

### 4.4 视频

保存抓取到的公开视频基础信息。

建议字段：
- `id`
- `bvid` 或其他视频唯一标识
- `up_id`
- `title`
- `video_url`
- `published_at`
- `duration`
- `cover_url`（可选）
- `fetched_at`
- `created_at`
- `updated_at`

### 4.5 视频本地状态

保存只属于用户本地的管理状态。

建议字段：
- `video_id`
- `watched`
- `watched_at`（可选）
- `created_at`
- `updated_at`

设计重点：
- 视频基础信息和用户本地状态分开存储
- 抓取更新不能覆盖 `watched` 等个人状态
- 标签属于 UP 主，不直接属于视频；标签页的视频集合由“标签下 UP 主的未看视频”推导得到

## 5. 页面设计

第一版只做三个页面。

### 5.1 标签视图

这是主使用页面。

功能：
- 展示标签列表
- 进入某个标签后，展示该标签下所有 UP 主的未看视频
- 默认按发布时间倒序
- 每条视频提供“打开 B 站视频”“标记已看”操作

每条视频卡片至少展示：
- 标题
- UP 主名
- 发布时间
- 时长

### 5.2 UP 主管理页

用于维护抓取源和分类。

功能：
- 添加 UP 主主页链接
- 编辑 UP 主名称
- 给 UP 主打多个标签
- 启用 / 停用同步
- 查看最近同步时间
- 对单个 UP 主执行手动同步

### 5.3 同步状态页

用于观察和管理同步任务。

功能：
- 展示最近一次全量同步时间
- 展示新增视频数量
- 展示失败的 UP 主及错误信息
- 手动触发立即同步
- 展示并配置定时同步状态与频率

## 6. 抓取与同步机制

### 6.1 抓取输入

抓取输入来源于用户手工录入的 UP 主主页链接。系统不从关注列表、收藏列表或观看历史自动导入。

### 6.2 抓取内容

第一版只抓页面展示必需的最小信息：
- UP 主标识
- 视频标题
- 视频链接
- 发布时间
- 时长
- 封面图（可选）
- 抓取时间

明确不抓：
- 个人观看记录
- 评论、弹幕、收藏、点赞等扩展信息
- 任何私有账号信息

### 6.3 同步方式

系统支持两种同步方式：

1. 手动同步：由用户在页面中主动触发。
2. 定时同步：由本地定时任务自动触发。

两种方式走同一套同步流程：
- 遍历启用中的 UP 主
- 抓取公开视频列表
- 进行去重和更新
- 记录同步结果

### 6.4 去重与更新规则

- 以视频唯一标识去重
- 新视频插入本地库
- 已存在视频更新公开元信息
- 用户本地状态（如 `watched`）不允许被抓取结果覆盖

### 6.5 失败处理

- 某个 UP 同步失败时记录错误信息和失败时间
- 单个 UP 失败不应中断整次同步
- 同步状态页需要展示失败结果，方便排查

### 6.6 风险控制原则

- 只抓公开页面
- 不依赖常用账号
- 控制同步频率
- 优先串行或低并发抓取
- 把稳定与低风险放在高吞吐之前

## 7. 系统结构

建议分成四层：

### 7.1 Web 界面层

职责：
- 展示标签视图、UP 主管理页、同步状态页
- 提供标记已看、立即同步等交互入口

### 7.2 应用服务层

职责：
- 查询某标签下的未看视频
- 更新视频已看状态
- 管理 UP 与标签的关联
- 触发同步任务并汇总结果

### 7.3 抓取层

职责：
- 根据主页链接获取公开视频列表
- 解析原始页面内容
- 输出标准化视频数据
- 返回抓取异常

### 7.4 本地存储层

职责：
- 持久化 UP、标签、关联关系、视频、本地状态、同步日志
- 提供稳定的数据读写能力

设计要求：
- Web 界面层不直接操作抓取逻辑
- 抓取层不感知标签或已看状态
- 应用服务层负责拼装业务语义
- 本地存储层不承载业务规则

## 8. 范围边界

### 8.1 第一版必须做到

1. 手动添加一个或多个 UP 主主页链接
2. 给每个 UP 主打多个标签
3. 手动同步这些 UP 主的公开视频列表
4. 本地定时同步
5. 标签页面展示对应标签下的未看视频
6. 手动将视频标记为已看
7. 已看视频不再出现在未看列表中
8. 同步状态页展示最近同步结果和失败信息
9. 所有个人数据都只保存在本地

### 8.2 第一版明确不做

- 不接入常用 B 站账号
- 不读取真实观看历史
- 不做笔记系统
- 不做首页聚合流
- 不做推荐算法
- 不做评论、弹幕、收藏等扩展数据
- 不做云同步和多端同步
- 不做移动端 App

## 9. 验收标准

如果用户能够稳定完成以下闭环，则第一版达到目标：

1. 录入一批 UP 主主页链接
2. 给这些 UP 主打标签
3. 通过手动或定时方式完成同步
4. 进入某个标签页
5. 看到该标签下所有未看视频
6. 看完后手动标记已看
7. 后续再次打开时，列表里只剩真正还没看的视频

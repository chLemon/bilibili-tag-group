# 标签创建补丁设计

## 1. 背景

当前 MVP 已支持：
- 查询标签列表
- 在创建或编辑 UP 主时选择已有标签
- 按标签查看未看视频

但系统缺少“创建标签”的入口。对于一个全新用户，数据库里没有任何标签时，无法完成“先建标签，再给 UP 主打标签”的主流程。这与现有设计文档中“给 UP 主打多个标签”的目标不一致，因此需要补一个最小可用补丁。

## 2. 目标

本次补丁只解决一个问题：让用户可以在现有 UP 主表单里直接创建标签，并立即把新标签用于当前表单。

补丁完成后，用户应能在“添加 UP 主”或“编辑 UP 主”表单中：
1. 输入新的标签名
2. 点击创建
3. 在当前表单里立即看到新标签
4. 新标签自动处于选中状态
5. 提交表单时把该标签关联到当前 UP 主

## 3. 非目标

本次不处理以下内容：
- 不新增独立的“标签管理页”
- 不支持删除标签、重命名标签、标签颜色等扩展能力
- 不修改标签页展示逻辑
- 不顺带补 `last_sync_at`、URL 校验、重复 `profile_url` 错误处理等其他评审项

## 4. 方案选择

### 方案 A：在 UP 主表单内联创建标签（采用）
- 后端新增 `POST /api/tags`
- 前端在 `CreatorForm` 中提供标签名输入框和创建按钮
- 创建成功后，把返回的标签加入当前可选列表并自动选中

优点：
- 直接补齐主流程，用户不需要离开当前表单
- 改动范围最小，复用现有标签加载和标签勾选逻辑
- 最符合本次“先补阻塞项再合并”的目标

缺点：
- `CreatorForm` 会增加少量局部状态和错误提示

### 方案 B：在 UP 主管理页放独立“新建标签”入口
优点：页面职责更清晰。
缺点：用户需要先建标签，再回表单选择，主流程更绕。

### 方案 C：只补后端 API
优点：后端改动最少。
缺点：前端仍不可用，不能真正解决当前阻塞项。

结论：采用方案 A。

## 5. 后端设计

### 5.1 新增 Schema
在 `app/schemas/tag.py` 中新增：
- `TagCreate`
  - `name: str`

仍复用现有 `TagRead` 作为返回体。

本次只做最小校验：
- 标签名去除首尾空白后不能为空

### 5.2 服务层
在 `app/services/tag_service.py` 中新增 `create_tag(db, name)`：
- 对输入名做 `strip()`
- 创建 `Tag` 记录并写入数据库会话
- 返回新建的 `Tag`

本次不额外设计复杂去重策略。若数据库唯一约束导致重复标签写入失败，先沿用当前项目的默认错误传播方式，不在本补丁中额外扩大处理范围。

### 5.3 路由层
在 `app/routers/tags.py` 中新增：
- `POST /api/tags`
- 请求体：`TagCreate`
- 响应体：`TagRead`
- 成功状态码：`201`

返回结果只包含：
- `id`
- `name`

## 6. 前端设计

### 6.1 API 客户端
在 `frontend/src/api/client.ts` 中新增：
- `createTag(payload: { name: string }): Promise<Tag>`

### 6.2 CreatorForm 交互
在 `frontend/src/components/CreatorForm.tsx` 中新增一段“创建标签”交互：
- 一个标签名输入框
- 一个“创建并选中”按钮
- 一个用于展示标签创建失败的内联错误提示

交互规则：
1. 输入为空白时，按钮禁用
2. 点击按钮后调用 `createTag`
3. 创建成功后：
   - 清空输入框
   - 把新标签加入当前标签列表
   - 自动把该标签加入 `tag_ids`
4. 创建失败后：
   - 保留输入内容
   - 在表单内联展示错误信息
5. 不影响原有“保存 UP 主”和“取消”按钮行为

### 6.3 状态归属
为了尽量减少改动，标签创建后的即时反馈状态放在 `CreatorForm` 内部维护：
- 本地维护一份可展示的标签列表，初始值来自 `props.tags`
- 当 `props.tags` 变化时，同步刷新本地列表

这样可以避免为了这次小补丁把 `CreatorsPage` 改成更大的全局状态协调结构。

## 7. 错误处理

本次只补最小必要错误处理：
- 标签名为空白：前端禁用按钮，后端拒绝空白名
- 创建标签接口失败：在表单内联提示

以下问题明确留到后续，不在本次补丁中展开：
- 重复标签名的友好错误文案
- 更严格的标签命名规则
- 跨页面的标签缓存同步策略

## 8. 测试设计

### 8.1 后端测试
在 `tests/test_routers.py` 中新增用例：
- `POST /api/tags` 成功返回 `201`
- 返回体包含新标签的 `id` 和 `name`

### 8.2 前端测试
在 `frontend/tests/CreatorsPage.test.tsx` 中新增用例：
- 打开“添加 UP 主”表单
- 输入新标签名并点击“创建并选中”
- mock `createTag` 成功返回新标签
- 断言新标签出现在表单中且复选框已选中

## 9. 验收标准

满足以下条件即可认为本补丁完成：
1. 新用户在没有预置标签的情况下，可以从 UP 主表单创建标签
2. 新建标签创建成功后，会立即出现在当前表单中
3. 新建标签会自动选中，并能随当前 UP 主一起提交
4. 原有标签查询、UP 主创建/编辑、标签筛选视频功能不回归
5. 新增后端与前端测试均通过

## 10. 对现有文档的影响

本补丁不改变产品边界，只是把已有设计目标补齐为可操作的最小实现。
后续在整理文档时，应把“用户可在 UP 主表单中直接创建标签”沉淀到正式产品/设计文档里，而不是单独长期保留这份补丁设计。
